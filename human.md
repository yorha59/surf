> 作用：本文件是 **当前迭代唯一的人类待办清单**，只记录「需要人类显式处理或决策」的问题。
>
> 使用约定：
> - 当任一 Agent（尤其是 Coco 在 Ralph-loop 中）判断「需要人类确认/介入」时，**必须在这里新增一条问题记录**，而不是继续往 `PRD.md` / `Architecture.md` 写 TODO。
> - 只要本文件**非空**，下一轮 Ralph 调用 Coco 时，应优先围绕这里列出的事项工作（例如检查人类是否已经按建议修改了代码/文档、更新状态等），在说明中显式反馈处理结果。
> - 当 Coco 认为本文件中的所有问题已经被人类处理完，且当前轮不再需要人类额外动作时，**应清空本文件内容（或只保留标题和说明段落）**，让后续 Ralph 迭代可以重新聚焦其他 story。

## 问题 1：JSON-RPC 服务的暴露范围与安全策略需确认

**背景：**
- `PRD.md` 中服务模式默认监听 `127.0.0.1:1234`，但同时提到 GUI 程序可以连接到“远程已存在的 server”；
- 当前文档未明确：
  - JSON-RPC 服务是否允许被局域网/互联网其他主机访问；
  - 是否需要身份认证、访问控制或加密（TLS）要求；
  - 在多用户环境（如共享服务器）下如何避免用户之间互相干扰。

**需要人类决策：**
1. Surf 的 JSON-RPC 服务在首版发布时，是否只支持本机访问（`127.0.0.1`），还是允许 GUI/其他客户端通过局域网访问？
2. 如果允许跨主机访问：
   - 是否需要强制启用认证（例如 token 或操作系统级身份绑定）？
   - 是否需要在 PRD 中增加关于加密传输（TLS 或通过隧道）的要求？
3. 是否需要在 PRD 中显式声明安全边界（例如“首版只支持本机访问，不适合作为多租户共享服务”）。

建议人类在 `PRD.md` 中补充一小节或段落，对 JSON-RPC 服务的暴露范围和安全策略做出明确约束，以便后续设计和实现阶段收敛方案。

## 问题 2：删除操作的统一行为与平台差异需确认

**背景：**
- `PRD.md` 中：
  - TUI 需求提到“支持键盘操作进行目录导航和文件删除”；
  - GUI 需求中右键菜单包含 `Move to Trash`；
  - 范围（Scope）部分强调核心扫描与分析为只读，删除能力作为单独操作能力提供；
- 当前文档未明确：
  - Linux 平台下的“删除”是永久删除还是有统一的“回收站”行为；
  - CLI / TUI / GUI 三种形态是否应遵循同一删除语义；
  - 是否允许目录级递归删除，以及是否需要额外的风险提示。

**需要人类决策：**
1. 首版中删除操作的统一策略是什么？例如：
   - macOS GUI：始终使用系统废纸篓（Move to Trash）；
   - Linux/TUI/CLI：默认永久删除，还是集成桌面环境的 Trash 机制，或只提供“打印建议，不实际删除”？
2. 是否要求所有形态（CLI/TUI/GUI）在删除前都进行统一的二次确认提示？
3. 是否需要在 PRD 中限制一次性可删除的最大体量或类型（例如禁止直接删除整块挂载点、系统目录等），以降低误操作风险？

建议人类在 PRD 中增加对删除行为的统一说明（按平台和形态分别说明），并明确安全策略与 UX 要求，以便核心引擎和各前端模块在实现时保持一致。
